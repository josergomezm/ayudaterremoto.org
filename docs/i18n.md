# i18n Cookbook

vue-i18n in Composition API mode (`legacy: false`). Spanish (`es`) is the primary authored locale (field language for Venezuela); English (`en`) is the fallback. Setup lives in `src/i18n/index.ts`.

Strings live in **per-feature TypeScript modules**, not in monolithic JSON. Each module co-locates both locales and types `en` against `es`, so a missing or extra key in either language is a **compile-time error** — not a runtime warning.

## Structure

```
src/i18n/
├── index.ts                 # assembles modules, builds messages, type augmentation
└── locales/
    ├── configs/             # cross-cutting vocab (common, roles, triage, category)
    ├── components/          # component-scoped strings (shell, offline, …)
    └── pages/               # page/route-scoped strings (map, report, verify, guides, …)
```

The folder is organizational only. Each module registers under its **namespace key** (the file name), so keys stay `t('report.title')` regardless of folder.

## Module format

```ts
const es = {
  title: 'Nuevo reporte',
  submit: 'Enviar reporte',
}

// Typed against es: a missing or extra key here is a compile error.
const en: typeof es = {
  title: 'New report',
  submit: 'Send report',
}

export default { es, en }
```

`es` is the source of truth. Because `en` is typed `typeof es`, you literally cannot ship a namespace where the two locales diverge — TypeScript rejects it. Arrays work too (e.g. the guides' `steps`/`avoid` lists): `en` must supply a `string[]` of the same key.

## 1. Add a translated string

1. Open the module for that feature (e.g. `locales/pages/report.ts`).
2. Add the key to **both** `es` and `en` in the same edit. If you forget one, `tsc` fails on `en: typeof es`.
3. In the component: `const { t } = useI18n()` then `t('report.key')`. Keys autocomplete from the schema.

Never options-API `$t`, never a hardcoded user-facing string in a template or store.

> Type-safety note: `en`↔`es` key parity is enforced at compile time, and key **autocompletion** works (via the `DefineLocaleMessage` augmentation in `index.ts`). However, vue-i18n keeps a permissive `t(key: string)` overload, so an outright typo in a `t()` call is *not* a `tsc` error — vue-i18n's missing-key console warning remains the dev-time backstop for that case.

## 2. Add a new namespace

1. Create `locales/<folder>/<name>.ts` exporting `{ es, en }` in the format above.
2. In `index.ts`, import it and add one line to each of the `es` and `en` blocks of `messages`. (If the namespace collides with a JS reserved word like `import`, alias the binding — e.g. `import importMessages from './locales/pages/import'` — but keep the registered key unchanged.)

## 3. Add a new locale

1. For every module, add the new locale alongside `es`/`en`, or restructure the `{ es, en }` exports to include it. Diff against `es` to translate everything.
2. Add the code to `SUPPORTED_LOCALES` and the `messages` blocks in `index.ts`, and to `datetimeFormats`.

## 4. Translate data (not just UI chrome)

Stores and seed data hold message **keys** where the content is app-owned — enum display names (`t('roles.' + role)`, `t('triage.' + status)`, `t('category.' + cat)`), guide ids, announcement category labels. Content typed by people in the field — report descriptions, proxy subject names, the body of a broadcast — is **user data** and is NOT translated.

## 5. Dates & numbers

Use `d(date, 'short')` / `n(value, …)` from `useI18n()` — never hand-format. Named datetime formats (`short`, `time`) are defined in `index.ts` under `datetimeFormats`.

## The rules

Every user-facing string is a key in a per-feature module, present in both `es` and `en`. Adding a string = both locales, same commit, enforced by the type system. Data stays translatable via keys. `LocaleSwitcher.vue` flips the global locale (it lives on the About page).
